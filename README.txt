## 1. نظرة عامة على المشروع

**محرك معرفة RHEL العربي** هو منصة عربية تعمل داخل المتصفح، وتجمع المعرفة والتعلم والتنفيذ الموجّه وتشخيص مشكلات Red Hat Enterprise Linux.

يمكنك تجربة المشروع مباشرة على الرابط التالي:
https://rhel-arabic-dictionary.pages.dev

المشروع ليس مجرد قاموس أوامر، بل يجمع بين:

- المهام العملية،
- مراجع الأوامر،
- مفاهيم Linux،
- أدلة حل المشكلات،
- مسارات التعلم،
- البحث العربي باللغة الطبيعية،
- مسارات التنفيذ الموجهة،
- أشجار Linux Doctor،
- وتحليل مخرجات الطرفية محلياً.

ملف التشغيل الرئيسي هو:

```text
index.html
```

المشروع موقع ثابت يمكن نشره على Cloudflare Pages أو GitHub Pages أو أي خادم ملفات ثابتة.

## 3. الهدف الرئيسي

```text
سؤال أو مشكلة بالعربية
          ↓
فهم النية والبحث الذكي
          ↓
مهمة أو أمر أو مفهوم أو صفحة تشخيص
          ↓
تنفيذ موجه
          ↓

          ↓
اكتشاف نمط الخطأ
          ↓
اقتراح الفحص التالي
          ↓
اقتراح المعالجة
          ↓

```

يمكن أن يستفيد منه:

- المبتدئون في Linux،
- متدربو RHEL،
- مسؤولو الأنظمة،

- المدربون،
- والمنشآت التي تحتاج إلى قاعدة معرفة عربية لأنظمة Linux.

## 4. أنواع كيانات المعرفة

يدعم المشروع خمسة أنواع رئيسية.

### 4.1 المهمة العملية

المهمة هي إجراء كامل مثل:

- تثبيت Nginx،
- إنشاء مستخدم،
- فتح منفذ،
- إعداد خدمة،
- أو إدارة التخزين.

يمكن أن تحتوي المهمة على:

- الهدف،
- الملخص،
- المتطلبات،
- المتغيرات،
- الخطوات المرتبة،
- الأوامر،
- شرح كل خطوة،
- النتيجة المتوقعة،
- تحذيرات السلامة،
- أوامر التحقق،
- الأخطاء الشائعة،
- خطوات التراجع،
- الملفات والمنافذ،
- والمواضيع المرتبطة.

### 4.2 مرجع الأمر

صفحة تشرح أمراً مثل:

```text
dnf
systemctl
journalctl
firewall-cmd
ss
```

وقد تشمل الصيغة والخيارات والأمثلة والاستخدامات والتحذيرات والمواضيع المرتبطة.

### 4.3 المفهوم

صفحة تشرح فكرة أو نظاماً داخل Linux مثل:

- systemd،
- SELinux،
- RPM،
- المستودعات،
- الصلاحيات،
- LVM،
- أنظمة الملفات،
- أو الشبكات.

### 4.4 حل المشكلة

صفحة تبدأ من مشكلة مثل:

- خدمة Failed،
- SSH لا يعمل،
- Permission denied،
- امتلاء القرص،
- أو تعذر الوصول إلى منفذ.

وتعرض الأسباب المحتملة والفحوصات والإصلاحات والتحقق.

### 4.5 مسار التعلم

مسار يرتب مجموعة من المواضيع والمهام والأوامر، ويحفظ تقدم المتعلم داخل المتصفح.

## 5. البحث العربي باللغة الطبيعية

محرك النية مصمم لفهم عبارات المستخدم العربية، وليس فقط التطابق الحرفي.

أمثلة:

```text
ابي اثبت nginx
الخدمة ما تشتغل
افتح بورت 8080
الهارد فل
ما اقدر ادخل SSH
سيلنكس مانع الخدمة
```

يمكن للمحرك:

- توحيد النص العربي،
- فهم المترادفات والتعبيرات الدارجة،
- فهم المصطلحات التقنية العربية والإنجليزية،
- تصحيح بعض الأخطاء الإملائية،
- تحديد النية المحتملة،
- استخراج الحزمة والخدمة والمنفذ والخادم والمستخدم والمجموعة والمسار،
- وترتيب النتيجة الأقرب.

الملفات المهمة:

```text
intent-engine.js
intents.json
```

## 6. Linux Doctor

هو نظام تشخيص تفاعلي مبني على شجرة قرارات.

يسأل المستخدم سؤالاً محدداً، ويعرض أمر فحص، ثم يغير المسار حسب الإجابة.

تشمل المجالات الحالية:

- فشل الخدمات،
- مشكلات SSH،
- امتلاء نظام الملفات،
- فشل تثبيت DNF،
- مشكلات الشبكة وDNS،
- ارتفاع الحمل،
- مشكلات الصلاحيات،
- عدم الوصول إلى المنافذ،
- ورفض SELinux.

خصائصه:

- لا ينفذ الأوامر.
- يحفظ الجلسة في المتصفح.
- يسمح بالرجوع.
- يعرض النتيجة المتوقعة.
- ينشئ تقرير تشخيص.

الملفات المهمة:

```text
doctor-engine.js
doctor-data.json
```

## 7. مسارات التنفيذ الموجهة

يمكن فتح المهام وصفحات التشخيص على شكل جلسة تنفيذ منظمة:

```text
الاستعداد
   ↓
المتغيرات المطلوبة
   ↓
خطوات التنفيذ
   ↓
تحليل مخرجات الطرفية
   ↓
التحقق
   ↓
نتائج التشخيص
   ↓
التراجع
   ↓
التقرير النهائي
```

يستطيع محرك المسارات:

- استبدال متغيرات مثل `<SERVICE>` و`<PACKAGE>` و`<PORT>`,
- عرض أمر واحد في كل مرة،
- شرح كل خطوة،
- عرض النتيجة المتوقعة،
- حفظ التقدم،
- إيقاف الجلسة واستكمالها،
- تسجيل نجاح أو فشل الخطوات،
- وتصدير تقرير نصي.

الملفات المهمة:

```text
workflow-engine.js
workflows.seed.json
```

## 8. محرك تحليل مخرجات التنفيذ

هذه هي الميزة التشخيصية الأساسية في النسخة الحالية.

يقوم المستخدم بـ:

1. نسخ الأمر من الموقع.
2. تنفيذه داخل طرفية RHEL.
3. لصق المخرجات كاملة في الموقع.
4. طلب تحليلها محلياً.

يحاول المحرك تحديد:

- مكان الفشل،
- نمط الخطأ،
- درجة الثقة،
- الدليل من المخرجات،
- الأسباب المحتملة،
- أمر الفحص التالي،
- المعالجة المقترحة،
- وأمر التحقق من نجاح الحل.

مثال:

```text
nginx: [emerg] bind() to 0.0.0.0:80 failed
(98: Address already in use)
```

يستنتج المحرك أن المنفذ مستخدم، ثم يقترح:

```bash
sudo ss -tulpn | grep ':80'
```

بعدها يمكن لصق نتيجة الفحص ومتابعة سلسلة التشخيص.

يغطي المحرك أنماطاً مثل:

- فشل خدمات systemd،
- الوحدات المفقودة أو Masked،
- تعارض المنافذ،
- أخطاء صياغة ملفات الإعداد،
- مشكلات الصلاحيات،
- رفض SELinux وAVC،
- نفاد المساحة وinodes،
- أنظمة الملفات للقراءة فقط،
- أخطاء DNS والشبكة،
- مصادقة SSH،
- مستودعات DNF والاعتماديات،
- أخطاء GPG،
- تعارض الحزم،
- Mount وLVM،
- OOM،
- أخطاء I/O،
- Segmentation fault،
- وحدود الملفات المفتوحة.

الملفات المهمة:

```text
execution-diagnostics.js
diagnostic-patterns.json
```

## 9. الخصوصية والتخزين المحلي

المشروع يعتمد مبدأ العمل المحلي.

لا يحتاج إلى:

- خادم خلفي،
- قاعدة بيانات مركزية،
- API ذكاء اصطناعي،
- اتصال مباشر بالطرفية،
- أو خدمة تشخيص خارجية.

يتم تحليل المخرجات باستخدام JavaScript داخل المتصفح.

يمكن أن يحفظ المتصفح:

- المفضلة،
- نمط العرض،
- تقدم التعلم،
- جلسات المسارات،
- جلسات Linux Doctor،
- المتغيرات،
- المخرجات الملصقة،
- وسجل التشخيص.

يتم الحفظ داخل `localStorage`.

> قد تحتوي مخرجات الطرفية على أسماء مستخدمين أو أجهزة أو عناوين IP أو مسارات أو نطاقات داخلية أو معلومات حساسة. لا يرفعها الموقع إلى خادم بحسب تصميمه، لكن يجب مراجعة التقارير قبل مشاركتها.

---

## 10. نموذج السلامة

### الموقع لا ينفذ الأوامر

يقوم الموقع فقط بـ:

- عرض الأوامر،
- نسخها،
- تحليل النص الملصق،
- واقتراح الفحص التالي.

يبقى المستخدم مسؤولاً عن تنفيذ كل أمر في الخادم والبيئة الصحيحة.

### التشخيص إرشادي

مطابقة الأنماط تستطيع فهم الأخطاء الشائعة، لكنها لا تفهم كل تطبيق مخصص أو اعتماد إنتاجي أو تصميم بنية تحتية أو سياسة أمنية.

قبل تنفيذ أي تغيير:

- اقرأ الأمر كاملاً.
- تأكد من الخادم المستهدف.
- افهم مستوى الخطورة.
- أنشئ نسخة احتياطية عند الحاجة.
- اختبر داخل مختبر أو نافذة صيانة.
- اتبع سياسة إدارة التغييرات في الجهة.

### مشروع مستقل

المشروع مرجع مستقل للتعلم والعمل التشغيلي، وليس منتجاً رسمياً أو قناة دعم رسمية من Red Hat.

### ترخيص المشروع

لا تحتوي الحزمة الحالية على ترخيص للمشروع ما لم تتم إضافة ملف `LICENSE` منفصل. يجب اختيار ترخيص واضح قبل إعادة التوزيع أو استقبال مساهمات خارجية.

## 13. تشغيل المشروع محلياً

يقرأ الموقع ملفات JSON باستخدام `fetch()`، لذلك قد لا يعمل فتح `index.html` مباشرة عبر `file://`.

شغل خادم HTTP من داخل مجلد المشروع.

### باستخدام Python

```bash
python -m http.server 8000
```

ثم افتح:

```text
http://localhost:8000/
```

يمكن استخدام أي خادم ملفات ثابتة بشرط أن يقدم جميع الملفات من جذر المشروع نفسه.

## 14. النشر

يمكن نشر المشروع على Cloudflare Pages أو GitHub Pages أو أي استضافة Static.

الخطوات العامة:

1. ارفع جميع الملفات إلى جذر المستودع.
2. ضع `index.html` في الجذر.
3. حافظ على ملفات JSON وJavaScript بجواره.
4. اضبط الاستضافة كموقع ثابت.
5. لا تضف خادماً خلفياً ما لم تظهر حاجة مستقبلية.
6. انشر المستودع.

لا يحتاج الموقع إلى متغيرات بيئة أو قاعدة بيانات خادم أو ترحيل بيانات.

## 16. إضافة قاعدة تشخيص

أضف القاعدة إلى:

```text
diagnostic-patterns.json
```

مثال مبسط:

```json
{
  "id": "address-in-use",
  "title_ar": "المنفذ مستخدم من عملية أخرى",
  "patterns": [
    "address already in use"
  ],
  "domains": [
    "service",
    "network",
    "port"
  ],
  "severity": "high",
  "base_confidence": 0.99,
  "explanation_ar": "شرح سبب المشكلة",
  "likely_causes_ar": [],
  "checks": [],
  "fixes_ar": [],
  "verification": []
}
```

الممارسات المقترحة:

- استخدم أنماطاً دقيقة.
- أضف أنماط استبعاد عند الحاجة.
- تجنب الكلمات العامة إلا في قواعد fallback.
- قدم فحوصات قراءة آمنة أولاً.
- افصل الفحص عن المعالجة.
- أضف أمر تحقق.
- أضف حالة اختبار.

## 20. ممارسات التطوير المقترحة

- حافظ على استقرار فرع الإنتاج.
- استخدم فرعاً مستقلاً للتغييرات الكبيرة.
- تحقق من JSON قبل كل Commit.
- أضف اختباراً لكل نية أو قاعدة تشخيص جديدة.
- استخدم إجراءات متوافقة مع RHEL.
- راجع الأوامر بحسب الإصدار.
- وضح خطورة الأوامر الحساسة.
- أضف التراجع والتحقق.
- لا تستخدم تعطيل SELinux كحل عام.
- لا تستخدم `chmod 777` كحل عام.
- افصل أوامر الفحص عن أوامر التغيير.
- راجع الصياغة العربية لتكون واضحة ومتسقة.

## 21. أفكار التطوير المستقبلية

- زيادة المهام العملية متعددة الخطوات.

## 22. إخلاء المسؤولية

المشروع مرجع تعليمي وتشغيلي.

يمكن لأوامر Linux تغيير حالة النظام أو إيقاف خدمات أو فتح منافذ أو تعديل صلاحيات أو حذف بيانات. افهم الأمر قبل تنفيذه، واتبع سياسات البيئة التي تعمل عليها.
